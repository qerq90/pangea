# Pangea — Архитектурный документ

> **Назначение.** Это целевая архитектура и «полярная звезда» проекта Pangea. Документ нужен, чтобы (1) сверяться, идём ли мы в правильную сторону, и (2) дать контекст будущему ассистенту/ревьюеру, который подключится к проекту без истории обсуждений. Документ описывает **куда мы движемся** и **где находимся сейчас** — это разные вещи, и они явно разведены (см. раздел «Статус: текущее vs целевое»).
>
> **Заметка будущему Claude/ревьюеру.** Прежде чем предлагать изменения — прочитай раздел «Принятые решения и обоснования» и «Чего НЕ строим». Многие «очевидные» альтернативы (event sourcing, кодоген всех сцен, состояние в payload кнопки) уже рассмотрены и сознательно отклонены под конкретные боли. Не возвращай их без новой причины.

---

## 1. Что такое Pangea

Текстовая RPG в виде чат-бота. Игрок взаимодействует через кнопки и текстовые сообщения; бот отвечает сообщениями. Сейчас интерфейс — ВКонтакте, в коде уже заложен и Telegram. Игровой цикл: регистрация → выбор расы → подземелье/лабиринт → случайные события (находки, бой) → инвентарь, статы, прокачка.

По своей сути это **конечный автомат над графом сцен** с доменной механикой (генерация лута, бой, статы) и случайностью.

## 2. Технологический стек

| Слой | Технология |
|---|---|
| Язык | Scala |
| Эффекты/конкурентность | ZIO (`Task = ZIO[Any, Throwable, A]`) |
| БД | PostgreSQL |
| Доступ к БД | doobie |
| JSON | circe |
| Перечисления | enumeratum (+ `DoobieEnum`) |
| Миграции | goose (SQL) |

DI — через конструкторы + `ZLayer` (сервисы инжектятся как поля, **не** через ZIO-окружение `R`). Этой конвенции придерживаемся везде.

## 3. Принципы (полярная звезда)

Семь обязательств, по которым проверяется любое изменение:

1. **Движок в центре, интерфейс — адаптер.** Игровая логика не знает про ВК. ВК/Telegram/БД/тесты — сменные адаптеры вокруг ядра.
2. **Контент — данные, механика — код.** Тексты, кнопки, переходы описываются декларативно; код остаётся только там, где есть логика (эффекты, доменные вычисления).
3. **Functional core / imperative shell.** Доменные решения (бой, лут, статы) — чистые функции без IO. Эффекты-shell их оркеструют: загрузил → позвал ядро → сохранил → отрендерил.
4. **Детерминизм через явный RNG.** Случайность — это вход (seed), а не скрытый `new Random()`. Ядро воспроизводимо; роллы пишутся в журнал.
5. **Три вида состояния — три хранилища.** Durable game state, live scene state, append-only журнал не смешиваются.
6. **Надёжность чат-бота by construction.** Сериализация действий на игрока + идемпотентность апдейтов — структурно, а не «по соглашению».
7. **Граф валидируется на старте.** Битые переходы, недостижимые узлы, тупики ловятся при загрузке, не у игрока.

## 4. Слои архитектуры

Фактическая раскладка по пакетам (`core/src/main/scala/pangea/…`):

```
domain/      — Rng (чистый детерминированный генератор, без ZIO)
generator/   — чистое ядро: item/, loot/, monster/ (явный Rng на входе)
model/       — доменные модели: hero, item, monster, battle, trauma, stats, state…
engine/      — движок сцен + порты: Screen, Choice, Beat, Branch, Narrative,
               Target, SceneContent, GraphValidator, PartitionManager,
               Renderer, Players, Journal (порты живут здесь, не в отдельном ports/)
service/state/        — State, StateHandler, PlayerLock, UserAction + states/ (узлы графа)
service/sender/vk/    — адаптеры: VkApi, VkRenderer, VkPlayers
dao/ , repository/    — доступ к Postgres (включая dao/journal — JournalLive)
resources/scenes.yaml — контент сцен (тексты + кнопки) как данные
app/                  — Main (сборка ZLayer), server/ (вебхук-сервер ВК)
```

Зависимости направлены внутрь: адаптеры (`service/sender/vk`, `dao`) → порты (`engine`) ← `service/state` → `generator`/`domain`/`model`. Ядро ни от чего наружу не зависит. Отдельных пакетов `ports/`, `content/`, `adapters/` нет — порты лежат в `engine`, контент в `resources/scenes.yaml`, адаптеры в `service/sender` и `dao`.

### 4.1. Доменное ядро (`domain` / `generator`)

Чистые функции, без ZIO/IO, RNG передаётся явно как `domain.Rng`:

```scala
case class Rng(seed: Long) {
  def nextLong: (Long, Rng)
  def between(min: Long, max: Long): (Long, Rng)   // [min, max)
  def pick[A](list: List[A]): (A, Rng)
}

// generator/ — все принимают Rng и возвращают (результат, Rng)
ItemGenerator.createItem(lvl: Long, rarity: Rarity, rng: Rng): (Item, Rng)
LootGenerator.roll(tier, race, killLevel, rng: Rng): (List[LootDrop], Rng)
MonsterGenerator.generate(dungeonLevel: Int, rng: Rng): (Monster, Rng)
```

Тестируется property-тестами (`generator/*Spec`). Генераторы очищены от скрытого `Random` — `Rng` инжектится на вход. **Бой пока не вынесен в чистое ядро:** `BattleState` остаётся в shell и катает роллы через ZIO `Random` (см. §12).

### 4.2. Движок сцен (`engine`)

Платформо-независимое представление экрана и ввода построено и используется всеми сценами:

```scala
case class Screen(text: String, choices: List[Choice])
case class Choice(id: String, label: String, data: Map[String, String] = Map.empty)
```

Ввод не вынесен в отдельный `sealed trait Input` — он приходит как `UserAction` (payload-кнопка или текст); `Branch` извлекает из payload ключ `action` и роутит. `Narrative` разворачивает цепочку `Beat`'ов в маршруты. Per-state матчей по enum'у `Action` на сцену больше нет: `Action`-энумы сохранились лишь как **кодеки payload-ключей** (строки в JSON кнопки), а диспетчеризацию ведёт `Branch` по строковому ключу.

### 4.3. Порты и адаптеры (`engine` / `service/sender/vk`)

Порты объявлены в `engine`, реализации — в адаптерах:

```scala
trait Renderer { def show(user: User, screen: Screen): Task[Unit] } // VkRenderer
trait Players  { def getDisplayName(user: User): Task[String] }      // VkPlayers
trait Journal  { def append(event: GameEvent): Task[Unit] }          // JournalLive
```

`VkRenderer` превращает `Screen` в сообщение ВК с клавиатурой; `VkApi` парсит входящий апдейт в `UserAction`. Сцены работают только через `Renderer`/`Screen` — `Keyboard`/`sendMessage` из них убраны, ВК больше не протекает в стейты. Telegram пока заглушка: `StateHandler.makeActionTelegram` есть, но отдельного рендерера/вебхука нет (использует `VkRenderer`).

## 5. Модель сцены

Узел графа — один из трёх видов:

- **Нарратив** — текст + одна кнопка «дальше» → следующий узел. (Линейные катсцены, как цепочка Travel в регистрации.)
- **Выбор** — текст + несколько кнопок, каждая ведёт в свою цель.
- **Эффект** — выполняет код и по результату выбирает следующий узел.

Цель кнопки (`Target`) — либо данные, либо код:

```scala
sealed trait Target
object Target {
  case class Goto(state: StateType)                                  extends Target // данные → ключ сцены
  case class Run(f: (User, UserAction, Renderer) => Task[StateType]) extends Target // код (эффект)
}
```

`Branch(routes: Map[String, Target], fallback: Target)` маршрутизирует по строковому ключу `action` из payload кнопки; `gotoTargets` отдаёт все `Goto`-цели для валидатора графа.

**Граница данные/код проходит по `Target`.** `Goto` сериализуем (держит ключ сцены), `Run` несёт функцию (эффект — всегда код). Сцена — «в основном данные с явными дырками под код».

**Правила для эффектов:**
- Сервисы (api, dao, repo) **захватываются** (эффект — метод стейта с этими полями), а не передаются в сигнатуру.
- В shell остаётся оркестрация (вызвать ядро, сохранить, отрендерить, выбрать следующую сцену) — это его законная работа, её не «прячут».
- Хелпера `Run.decoded[A]` (декод payload в доменный тип на границе) пока нет — эффекты, которым нужен payload (напр. выбор расы), читают `ua.payload` внутри себя. Это кандидат на будущее упрощение.

## 6. Модель состояния

| Вид | Что | Хранилище | Жизненный цикл |
|---|---|---|---|
| **Durable game state** | статы, инвентарь, прогресс | мутабельные таблицы (`heroes`, `inventory`) | источник истины |
| **Live scene state** | текущая сцена + scratchpad (выкаченный предмет до клика) | одна колонка-блоб на герое (`scene_data JSONB`) | перезаписывается следующей сценой |
| **Журнал** | что произошло (роллы, исходы, ошибки) | append-only таблица, партиционированная | retention-окно |

**Payload кнопки несёт только ссылку** (какая сцена / какой выбор), не состояние. У ВК payload ограничен по размеру; состояние живёт на сервере (live scene state), в кнопке — крошечный идентификатор.

Инвариант «ровно одно живое состояние на игрока» гарантирован **структурно** (одна колонка одной строки героя), а не соглашением.

## 7. Детерминизм и RNG

Игра полна случайности (`rollLoot`, выбор события, бой). RNG — инжектируемый seed на вход чистого ядра. Это даёт:
- тестируемость ядра без моков;
- воспроизводимость багов: seed и реальные роллы пишутся в журнал → сессию можно переиграть как было.

Запрет: скрытые `new Random()` внутри доменной логики.

## 8. Надёжность чат-бота

> **Статус: ✅ реализовано.** Чат-платформы передоставляют сообщения, игроки жмут дважды, апдейты приходят внахлёст. Решено структурно:

- **Сериализация по игроку.** `PlayerLock` — `Ref[Map[VkId, Semaphore]]`, по семафору на игрока (`withLock`). `StateHandler.makeActionVK` оборачивает всю обработку апдейта в `lock.withLock(vkId)` → переходы упорядочены, не гоняются. By construction убивает класс «два конкурентных перехода».
- **Идемпотентность.** Апдейт несёт `eventId`; `userRepo.checkAndRecordEvent` атомарно пишет `users.last_event_id` через `UPDATE … WHERE last_event_id IS NULL OR last_event_id <> $eventId` и возвращает, новый ли это апдейт. Повтор того же `eventId` → `isNew = false`, обработка пропускается. Дабл-тап не ломает стейт.

## 9. Журнал и партиционирование

> **Статус: ✅ реализовано.** Порт `engine.Journal` + `dao/journal/JournalLive` пишут в партиционированную `event_log` (миграция `20260617000000_event_log.sql`). `engine.PartitionManager.ensurePartitions` нарезает партиции на 3 месяца вперёд при старте `JournalLive`. `GameEvent` пишется в ключевых точках: `race_selected` (Registration), `item_found`/`item_left` (FoundItem), `loot_claimed` (Loot). Старая `events`-таблица как live-state выведена из обращения — live-состояние теперь в `heroes.scene_data`.

Журнал — **append-only**: только `INSERT`, никаких `status`/`UPDATE`/`.unique`. Строка — это факт, который случился.

```sql
CREATE TABLE event_log (
    id BIGSERIAL, user_id BIGINT NOT NULL,
    event_type VARCHAR(64) NOT NULL, payload JSONB NOT NULL,
    trace_id UUID, occurred_at TIMESTAMP NOT NULL DEFAULT now(),
    PRIMARY KEY (id, occurred_at)          -- ключ партиции обязан входить в PK
) PARTITION BY RANGE (occurred_at);
```

- Партиции по месяцу; `DEFAULT`-партиция как предохранитель; нарезка на пару месяцев вперёд.
- Ретеншн = `DROP TABLE` старой партиции (мгновенно), а не `DELETE`.
- Обслуживание: **pg_partman (+pg_cron)**, если хостинг даёт расширения; иначе ZIO-джоб «создать вперёд / дропнуть старое». Решает доступность расширений на сервере.
- `trace_id` связывает строки одного действия в цепочку.
- Запросы — с границей по `occurred_at`, иначе нет partition pruning.

Логировать стоит **роллы и исходы** (что выкатил генератор, влез ли предмет, что выбрал игрок, ошибки) — то, что иначе не восстановить.

## 10. Жизненный цикл апдейта

1. ВК-вебхук → адаптер парсит в `Update(playerId, Input)`.
2. Диспетчер отдаёт в **сериализованный по игроку** обработчик.
3. Загрузка игрока (durable + текущая сцена + scratchpad).
4. Глобальная команда (`/home`, `/stats`) → обработать; иначе `Tap` → найти выбор в текущей сцене → выполнить `Target` (`Goto`/эффект).
5. Эффект: вызовы ядра + персист + `journal.append`, возвращает следующую сцену.
6. Рендер сцены через `Renderer` → ВК-адаптер шлёт сообщение.
7. Сохранить позицию + scratchpad; пометить апдейт обработанным.

## 11. Тестирование

- **Ядро** — property-тесты (инварианты боя/лута/статов).
- **Граф сцен** — headless-драйвер: «бот-игрок» ходит по графу через `Input`/`Screen` без ВК; целые нарративные ветки проверяются юнит-тестами.
- **Журнал** — проверка, что в ключевых точках падают правильные события.
- **Старт** — валидатор графа как тест: нет битых переходов/недостижимых узлов.

## 13. Путь миграции — что сделано и что осталось

Исходный инкрементальный план почти полностью пройден. Состояние шагов:

1. ✅ **Движок рядом с `State`.** `Branch`/`Narrative`/`Target` встроены за `State`-интерфейсом, схему БД не ломали.
2. ✅ **`Registration` первой.** Цепочка Travel → `Narrative.toRoutes`, вкачено.
3. ✅ **`Dungeon`, `HeroStats`, выбор расы** → узлы-выборы (`Branch` + `Goto`/`Run`).
4. ✅ **`FoundItem`** → `enter`-эффект + кнопки-`Run`; логика инвентаря не менялась.
5. ✅ **Состояние событий.** `heroes.scene_data` (live) + `event_log` (append-only журнал); старая `events`-таблица выведена из обращения.
6. 🟡 **Очистить ядро.** `generator/*` на явном `Rng`. **Осталось:** вынести бой из `BattleState` в чистое ядро на `Rng`.
7. ✅ **Вынести интерфейс в адаптер.** `Screen`/`Renderer` введены, ВК-специфика убрана из стейтов; `VkRenderer`/`VkPlayers` — адаптеры.
8. 🟡 **Контент в ресурсы.** `scenes.yaml` + `SceneContent` подключены ко всем сценам; часть динамических текстов ещё в Scala. i18n пока нет.
9. ✅ **Надёжность.** `PlayerLock` (per-user сериализация) + `checkAndRecordEvent` (дедуп апдейтов).

**Осталось (новый бэклог):**
- Бой как functional core с явным `Rng` (журналирование роллов боя как следствие).
- `Run.decoded[A]` — декод payload в доменный тип на границе эффекта.
- Сквозные headless-тесты флоу (бот-игрок по графу через `Input`/`Screen`), а не только per-state `*Spec`.
- Полноценный Telegram-адаптер (свой рендерер + вебхук), а не заглушка на `VkRenderer`.
- Догнать контент: оставшиеся динамические тексты в `scenes.yaml`.

Порядок такой, что каждый шаг — самостоятельный PR с откатом.

## 14. Принятые решения и обоснования

| Решение | Почему | Что отклонено |
|---|---|---|
| Сцены как данные + интерпретатор | Контент в коде (Travel1..9) — 10 почти одинаковых методов; вставка сцены = renaming | Кодоген всех сцен в Scala (сотни файлов против одного движка) |
| Эффект указывает на цель напрямую (`Run`) | `Action.X => method` — 4 прыжка, непрозрачно; enum-тег врёт (`Travel` = «подтвердили расу») | Сохранение per-state Action-enum |
| Сервисы захватываются, не передаются | Эффект — метод стейта с полями-зависимостями; сигнатуру не раздуваем | Тащить сервисы в ZIO-окружении `R` (расходится с DI проекта) |
| Декод транспорта на границе | `selectRace`, дёргающий `ua.payload`, смешивает wire и логику | Эффекты, принимающие `UserAction` |
| Live state на герое + append-only журнал | `events` с `.unique` + всегда-INSERT — латентный краш на дублях; `endEvent` без фильтра статуса | Одна `events`-таблица как live-state и лог одновременно |
| Явный RNG | Скрытый `new Random()` не воспроизвести; баг рандома не отладить | RNG внутри доменных функций |
| Движок в центре, ВК — адаптер | ВК протёк в стейты; флоу нетестируем; Telegram дублировался бы | Логика, завязанная на `Keyboard`/`sendMessage` |
| Партиционирование журнала сразу | Append-only растёт быстро; ввести партиции потом — больно | `DELETE` по дате (бloat + vacuum) |

## 15. Чего НЕ строим (YAGNI)

- **Event sourcing.** Журнал наблюдательный; истина — в мутабельных `heroes`/`inventory`. Полный ES — оверкилл.
- **Микросервисы.** Один сервис + Postgres держит большую аудиторию.
- **Свой DSL контента на старте.** Сначала данные (HOCON/ресурсы) + валидатор; DSL — когда заболит.
- **Распределённая сериализация игроков.** Внутрипроцессной хватит, пока не упрёшься.
- **Кодоген всего.** Целевой путь для flow — интерпретатор сцен (`Branch`/`Narrative`), а не генерация Scala. Временный кодоген клавиатур (`kbgen`) в проекте отсутствует.

## 16. Боевая механика и формулы

### Урон игрока
```
damage = (atk + str) × spread / 100 × weaponMod × concBonus
```
- `atk` — `fightStats.atk`, растёт от оружия и баффов
- `str` — `baseStats.str`, растёт от прокачки
- `spread` — равномерное случайное значение 80–120 (разброс ±20%)
- `weaponMod` — 0.5 без оружия, 1.0 с оружием
- `concBonus` — `1 + concentration × 0.005`

**Важно:** формула аддитивна (`atk + str`), не мультипликативна. Умножение (`atk × str`) давало квадратичный рост и было исправлено.

### Броня
```
maxArmor = equipment.allArmor × max(fightStats.defence, 1)
```
- `equipment.allArmor` — сумма `armor` всех надетых предметов
- `defence` — из предметов; при `defence = 0` множитель равен 1 (броня работает без вложений в защиту)
- В бою броня поглощает урон первой и расходуется; на отдыхе восстанавливается до `maxArmor`
- Пересчёт `fightStats` при надевании/снятии экипировки (`InventoryState.applyDelta`) зажимает каждую характеристику (`atk`, `armor`, `defence`, `evasion`, `accuracy`, `concentration`) в `.max(0L)` — снятие предмета не уводит текущие значения в минус

### Предметы и их ID
Каждый предмет получает реальный `BIGSERIAL` ID в таблице `items` **в момент генерации** (через `ItemRepository.generate` / `ItemRepository.persist`), до попадания в инвентарь. В JSON-блобе инвентаря и слотах экипировки хранится предмет с уже настоящим ID. Это гарантирует уникальность при `removeItem`.

### Отдых
После истечения `restDurationMs` (30 с — обычный, настраивается `DeathState`) восстанавливаются:
- HP до `effectiveMaxHp` (`vit × 16 × raceFactor`)
- Броня до `maxArmor`

### Боевой экран
Во время боя игроку показывается:
```
⚔ <Редкость Раса>
Раса: <раса> | ❤ <текущий>/<макс> | 🧥 <броня>/<макс> | ⚔ <урон>
🎯 Попадание: <шанс попасть по игроку>% | 💨 Уворот: <шанс увернуться от игрока>%

🧙 Вы: ❤ <текущий>/<макс> 🧥 <броня>/<макс>
🧪 Фляга: <заряды>
```
- **Попадание** — `mobHitChance(agi, evasion, battle)`, зажато в [5%, 95%]
- **Уворот** — `1 − playerHitChance(accuracy, battle)`, то есть вероятность, с которой моб уйдёт от удара игрока; зажато в [5%, 95%]
- Оба числа считаются из **эффективных** характеристик героя (с учётом баффов/дебаффов раунда)
- Броня показывается как `текущая/максимальная` и для моба, и для героя

### Фляга

Фляга — особый предмет (`ItemType.Flask`). Разные фляги выпадают с разными эффектами и запасом зарядов; логика эффекта закодирована прямо на предмете.

**Модель:**
```
Item.flaskEffect: Option[FlaskEffect]   // эффект при употреблении
Item.charges:     Option[Int]           // текущие заряды
Item.maxCharges:  Option[Int]           // максимум зарядов
```

**Эффекты (`FlaskEffect`):**
| Тип | Описание |
|-----|----------|
| `HealPercent(percent)` | восстанавливает `max(maxHp × percent / 100, 1)` HP |
| `AddBuff(buff, rounds)` | добавляет боевой бафф на `rounds` ходов |

**Правила боя:**
- Фляга должна быть экипирована в слот flask; без экипировки UseFlask возвращает ошибку
- **Один раз за раунд** — повторное нажатие заблокировано флагом `ActiveBattle.flaskUsedThisRound`; он сбрасывается в `tickBuffs`, который вызывается когда моб атакует в ответ на удар игрока
- Моб **не** наносит ответный удар — фляга бесплатное действие внутри хода

**Пополнение зарядов:**
- У источника воды (`DungeonState.healAtSpring`) восстанавливаются до `maxCharges` экипированная фляга и все фляги в инвентаре (`InventoryRepository.refillFlasks`)
- При создании героя стартовая фляга уже имеет `charges = maxCharges`, заданные на предмете — никакой глобальной константы нет

### Событие «Ручеёк» (Spring)

- HP полностью восстанавливается до `effectiveMaxHp`
- Все фляги (в слоте и в инвентаре) пополняются до `maxCharges`
- **50% шанс засады**: после лечения бросается `Random.nextIntBetween(1, 101)`; при результате ≤ 50 — немедленно стартует бой (`startBattle`), игроку показывается сообщение «Однако на шум живительного ручья пришёл и кто-то ещё…»

## 17. Опыт и уровни

### Модель

`hero.exp` хранит опыт **внутри текущего уровня** — не суммарный накопленный. При достижении порога уровня exp сбрасывается в 0, уровень растёт.

```
neededExp = lvl × 100        // порог для уровня 1 → 100, уровня 2 → 200, ...
```

Отображение: `exp / neededExp` (например, `37/100` на уровне 1).

### Level-up

Происходит в `BattleState.victory` через цикл, который обрабатывает несколько уровней за раз:

```scala
while (exp >= lvl * 100L && lvl < 150L) {
  exp  -= lvl * 100L
  lvl  += 1L
  upgradePoints += 4L
}
```

Каждый новый уровень даёт **+4 очка прокачки**. Максимальный уровень — 150.

### Смерть и опыт

Смерть **не снижает уровень**. Теряется 10% опыта внутри текущего уровня (`(exp × 0.1).max(0)`), уровень и очки прокачки остаются неизменными. Дыра «умереть на пороге уровня ради двойных очков» закрыта этой моделью: уровень не откатывается, повторно получить очки за тот же уровень невозможно.

## 18. Механика смерти

Смерть наступает в `BattleState`, когда HP героя падает до 0. Переход: `StateType.Battle → StateType.Death → StateType.Rest`.

**Состояния без пользовательского взаимодействия.** Состояние может полностью отыграться в `enter` и сразу перейти дальше, не требуя нажатия кнопки. Это даёт общий механизм `State.autoAdvance: Option[StateType]`: если игроку нечего решать, состояние указывает следующее, а `StateHandler.transitionTo` чейнит переход (persist + `enter`) без действия игрока. Цикл авто-переходов ограничен предохранителем (fuel); обычные узлы-выборы оставляют `autoAdvance = None`.

`Death` — **пример такого узла-эффекта**: его `enter` сразу начисляет штрафы, выбирает травму, роняет предметы и пишет `restDurationMs`, после чего через `autoAdvance = Some(Rest)` без кнопки «Продолжить» уходит в `Rest`.

### Штрафы при смерти

| Параметр | Формула |
|---|---|
| Опыт | −10% опыта **внутри текущего уровня** (`exp × 0.1`, min 0); уровень не снижается |
| Золото | −50% (`gold / 2`) |
| Травма | случайная из текущего тира (см. ниже); добавляется к списку активных травм |

### Механика травм

Каждая смерть добавляет **одну** новую травму. Травмы **накапливаются списком** и действуют одновременно, их штрафы суммируются по каждому стату. Таймер 8 часов от **последней смерти** сбрасывается при каждой смерти; когда он истекает — весь список обнуляется разом.

**Прогрессия по тирам:** пока не собраны все 4 лёгкие → выпадают только лёгкие; потом только средние (пока не собраны все 4); потом только тяжёлые.

#### Лёгкие травмы

| Травма | Штрафы |
|---|---|
| Ушиб ноги | −5% уклонение |
| Раздробленный палец | −5% урон, −5% сила |
| Ноющее ребро | −5% HP, −5% телосложение |
| Порванное ухо | −5% точность, −5% сосредоточение |

#### Средние травмы

| Травма | Штрафы |
|---|---|
| Порезанное ахиллесово сухожилие | −25% уклонение, −15% броня, −15% защита |
| Проколотая рука | −25% урон, −25% сила |
| Глубокий порез живота | −25% HP, −25% телосложение, −15% броня, −15% защита |
| Сломанный нос | −25% точность, −25% сосредоточение |

#### Тяжёлые травмы

| Травма | Штрафы |
|---|---|
| Повреждение коленной чашечки | −50% уклонение, −15% броня, −15% защита |
| Сломанная рука | −50% урон, −50% сила |
| Глубокий разрез правого бока | −50% HP, −50% телосложение |
| Расколотый череп | −50% точность, −50% сосредоточение, −75% интеллект |

**Хранение:** `heroes.trauma_names TEXT` (имена через запятую) + `heroes.trauma_until BIGINT`. Список активных травм читается через `Hero.activeTraumas(nowMs)`, суммарный штраф — через `Hero.combinedPenalties(nowMs)`. Штрафы применяются к каждому стату независимо в `effectiveFightStats` и `effectiveMaxHp`. Снятие — в таверне (`GlobalMapState`), обнуляет оба поля.

### Дроп предметов

Каждый предмет в **инвентаре** (не в слотах экипировки) имеет **25% независимый шанс быть потерянным**.

- Реализовано в `DeathState.dropItems`: `Random.nextIntBetween(0, 4)`, дроп при `roll == 0`
- Потерянный предмет удаляется из инвентаря (`InventoryRepository.removeItem`)
- Игроку показывается сообщение: **«{monsterName} забрал(а) {itemName} себе в качестве трофея.»**
- Имя монстра читается из `ActiveBattle` (через `heroDao.readActiveBattle`) **до** вызова `heroDao.clearActiveBattle`; если активной битвы нет — fallback `"Монстр"`

### Отдых после смерти

Время мёртвого режима растёт по **уровню героя** и асимптотически стремится к 90 минутам, не достигая их:

```
deathTimeMinutes = 90 − 89 × exp(−k × (lvl − 1))      // k = DeathState.RestGrowthK = 0.01
restDurationMs   = deathTimeMinutes × 60 000
```

- На 1 уровне — ровно **1 минута** (`90 − 89×exp(0) = 1`).
- С ростом уровня время монотонно увеличивается, но строго меньше 90 минут (асимптота).
- `k` (`DeathState.RestGrowthK`) задаёт скорость роста.

Записывается в `scene_data`; `RestState` читает это значение и блокирует выход до истечения времени.

## 19. Инвентарь и экипировка

### Слоты экипировки

`Equipment` содержит 14 именованных слотов:

| Слот | Поле | Тип |
|---|---|---|
| Шлем | `helmet` | `Helmet` |
| Наплечники | `shoulderPads` | `ShoulderPads` |
| Нагрудник | `chestPlate` | `ChestPlate` |
| Браслеты | `bracelets` | `Bracelets` |
| Перчатки | `gloves` | `Gloves` |
| Штаны | `pants` | `Pants` |
| Сапоги | `boots` | `Boots` |
| Амулет | `amulet` | `Amulet` |
| Кольцо 1 | `firstRing` | `Ring` |
| Кольцо 2 | `secondRing` | `Ring` |
| Пояс | `belt` | `Belt` |
| Фляга | `flask` | `Flask` |
| Оружие | `weapon` | `Weapon` |
| Доп. оружие | `additionalWeapon` | `AdditionalWeapon` |

Экипировка хранится как JSONB в колонке `heroes.equipment`.

### Два слота под кольца (авто-логика)

`ItemType.Ring` — единственный тип с двумя слотами. Логика надевания в `InventoryState`:

- `firstRing` свободен → кольцо идёт в `firstRing`
- `firstRing` занят, `secondRing` свободен → кольцо идёт в `secondRing`
- Оба заняты → кольцо идёт в `secondRing`, старое кольцо из `secondRing` возвращается в инвентарь

Игрок явно управляет кольцами через экран снаряжения (`EquipmentState`), где `firstRing` и `secondRing` — отдельные слоты с кнопкой «Снять».

### Экран инвентаря (`InventoryState`)

Пагинация по предметам в инвентаре. На каждом предмете:
- Кнопки «Надеть», «Выбросить», навигация, «Выйти»
- Для колец в шапке показываются оба текущих слота («Слот 1: X / Слот 2: свободен»)
- Шапка включает количество золота (`💰 N`)
- Надевание: `updateEquipmentAndFightStats`, вытесненный предмет → `addItem` в инвентарь

### Экран снаряжения (`EquipmentState`)

Пагинация по 14 слотам (`EquipmentState.slots`). Для каждого:
- Показывает имя слота, название предмета и его статы, или «Пусто»
- Кнопка «🔓 Снять» видима только если слот занят
- Снятие: `addItem` → если инвентарь полон, показывает ошибку и предмет остаётся
- После снятия пересчитываются `fightStats` через `InventoryState.applyDelta`
- Страница сохраняется в `heroes.scene_data` (аналогично `InventoryState`)

Доступ: `HeroStatsState` → кнопка «Снаряжение» → `StateType.Equipment`.

### Золото

Отображается в шапке экрана инвентаря (`📦 Инвентарь (N/M) | 💰 G`) и в экране пустого инвентаря. В экране снаряжения и на экране найденного предмета (`FoundItemState`) золото не показывается.

### Распределение очков характеристик (`HeroStatsState`)

Кнопка «Распределить характеристики» появляется только при `upgradePoints > 0`.

После нажатия открывается меню с четырьмя кнопками (+СИЛ / +ТЕЛО / +ЛОВ / +ИНТ) и кнопкой «Назад». Применение одного очка **не закрывает меню**: `applyUpgrade` сохраняет изменение, показывает подтверждение («Характеристика улучшена!»), затем вызывает `showUpgradeScreen` повторно. Та перечитывает героя и:
- если очки ещё есть — показывает меню снова с обновлённым счётчиком;
- если очки исчерпаны — показывает «Нет свободных очков» и кнопки стандартной навигации.

Игрок выходит явно через «Назад» или автоматически когда очки заканчиваются.

## 20. Дроп с врагов

> **Статус: ✅ реализовано.** При победе с моба падает лут из общего пула. «Что выпало» считает чистое ядро `LootGenerator` (`generator/loot`, явный `Rng`); начисление и показ — shell (`BattleState.victory` → промежуточный `LootState`). Это functional core / imperative shell (§3).

### Поток

1. `BattleState.victory` начисляет опыт/уровень (как раньше), затем катает лут `LootGenerator.roll(монстр.rarity, монстр.race, killLevel, Rng(seed))`. `killLevel = hero.dungeonLevel`.
2. Непросохранённые предметы (id = −1) + список золотых выпадений кладутся в `scene_data` (`LootState.LootData`), `victory` возвращает `StateType.Loot`.
3. `LootState.enter` («осматривая добычу, вы находите следующее:») персистит каждый предмет (реальный id) и кладёт в инвентарь; начисляет золото; показывает итог и кнопку «Продолжить» → `Dungeon`.
4. Полный инвентарь → предмет **теряется**, в списке строка «{name} — Инвентарь переполнен». Пустой лут → «вы не находите ничего ценного».

**Золото теперь не гарантировано** — старая постоянная награда `goldGained` убрана из `victory`, золото падает только как категория лута (решение игрока, ответ 3).

### Сколько слотов дропа катаем (по тиру моба)

Каждый слот срабатывает независимо со своим шансом (`LootGenerator.dropChances`).

| Тир | 1-й слот | 2-й | 3-й |
|---|---|---|---|
| Обычный | 30% | — | — |
| Необычный | 60% | 30% | — |
| Редкий | 100% | 60% | — |
| Мифический | 100% | 60% | — |
| Легендарный | 100% | 100% | 60% |

**Без повторов категории** (ответ 4): уже выпавшая категория исключается из следующих слотов.

### Веса категорий (если слот сработал)

Один взвешенный бросок 0–100. Веса **одинаковы для всех тиров** и в сумме дают 100 — на первом слоте «пусто» нет, слот всегда что-то роняет. На последующих слотах уже выпавшая категория исключается (без повторов, ответ 4), суммарный вес активных категорий падает, и появляется доля «пусто» (например, на 3-м слоте легендарного, когда экипировка и трофей уже выпали, остаётся только золото 26%, остальные 74% — «пусто»).

| Категория | Вес |
|---|---|
| Экипировка | 35 |
| Трофей | 39 |
| Груда золота | 26 |

«Груда золота» даёт золото по формуле `killLevel×4 ±20%` (мин 1).

### Редкость выпавшей экипировки (по тиру моба)

Маппинг рангов: серая→`Gray`, белая→`White`, зелёная→`Green`, синяя→`Blue`, фиолетовая→`Purple`, пурпурная→`Violet`, легендарная→`Orange`. Веса в `LootGenerator.gearRarityWeights` хранятся в долях `1/1 000 000` (фракции из исходного спека), суммы доведены до 100% (числовые ошибки спека исправлены: Common Green 16.4995; Rare/Myth/Leg добирают недостающие 0.5% в Green).

| Редкость | Обыч | Необ | Редк/Миф | Лег |
|---|---|---|---|---|
| серая (Gray) | 45 | 30 | 20 | 0 |
| белая (White) | 35 | 45 | 40 | 20 |
| зелёная (Green) | 16.4995 | 19.4995 | 31.0 | 31.0 |
| синяя (Blue) | 2 | 3 | 5 | 25 |
| фиолетовая (Purple) | 1 | 2 | 3 | 13 |
| пурпурная (Violet) | 0.5 | 0.5 | 0.5 | 9.5 |
| легендарная (Orange) | 0.0005 | 0.0005 | 0.5 | 1.5 |

**Уровень** выпавшей экипировки — текущий разброс `−0..6` из `ItemGenerator` (ответ 5): редкость передаётся в `ItemGenerator.createItem(killLevel, rarity, rng)`, который сам применяет разброс уровня.

### Трофеи

Хранятся **как обычный предмет в инвентаре** (ответ 1): `ItemType.Trophy`, без статов, с полями `lvl = killLevel` и `race = Some(монстр.race.entryName)` (новое поле `Item.race: Option[String]`, обратносовместимо — старый JSON декодится в `None`). Имя — `«<тип>» (<раса>) ур-<killLevel>`. Трофей **не экипируется** (нет в `attackItems`/`defenceItems`; `InventoryState` блокирует «Надеть» сообщением `inventory.notEquippable`). «Коэффициент трофея» из спека не реализован (назначение не определено — отброшен).

Тип трофея катается по весам (`LootGenerator.trophyWeights`, суммы = 100):

| Тип | Обыч | Необ | Редк | Миф | Лег |
|---|---|---|---|---|---|
| Реликвия | 1 | 5 | 15 | 15 | 25 |
| Талисман | 9 | 15 | 20 | 25 | 45 |
| Голова | 20 | 20 | 25 | 30 | 20 |
| Мешок с пожитками | 70 | 60 | 40 | 30 | 10 |

### Ключевые файлы

- `generator/loot/LootGenerator.scala` — чистое ядро (`roll`, `LootDrop`, все таблицы).
- `service/state/states/LootState.scala` — промежуточный экран добычи (`LootData` в `scene_data`).
- `service/state/states/battle/BattleState.scala` — `victory` катает лут и уходит в `Loot`.
- `model/item/Item.scala` (`race`), `model/item/ItemType.scala` (`Trophy`), `model/state/StateType.scala` (`Loot`).

Дроп пишется в журнал (`loot_claimed`: золото + имена предметов), как и `FoundItemState`.

## 21. Глоссарий

- **Сцена (Scene/State)** — узел графа: что показать и какие переходы. Бывает нарратив, выбор, эффект.
- **Бит (Beat)** — одна реплика линейного нарратива: текст + кнопка «дальше».
- **Эффект (Effect)** — код, выполняемый по кнопке: оркестрирует ядро/сервисы и выбирает следующую сцену.
- **Цель (Target)** — куда ведёт кнопка: `Goto` (переход-данные) или `Run` (эффект-код).
- **Live scene state / scratchpad** — переходное состояние текущей сцены (блоб на герое).
- **Durable game state** — постоянные данные игрока (статы, инвентарь).
- **Журнал (event_log)** — append-only лог фактов для дебага/аналитики.
- **Functional core / imperative shell** — чистое доменное ядро vs эффект-оркестратор.
- **Renderer / Adapter** — порт показа `Screen` и его реализация под платформу (ВК/Telegram/тест).
- **`EquipmentState.Slot`** — описатель слота экипировки: имя, геттер `get: Equipment => Item`, сеттер `clear: Equipment => Equipment`.
- **trace_id** — идентификатор, связывающий записи журнала одного действия.
