# Pangea — Архитектурный документ

> **Назначение.** Это целевая архитектура и «полярная звезда» проекта Pangea. Документ нужен, чтобы (1) сверяться, идём ли мы в правильную сторону, и (2) дать контекст будущему ассистенту/ревьюеру, который подключится к проекту без истории обсуждений. Документ описывает **куда мы движемся** и **где находимся сейчас** — это разные вещи, и они явно разведены (см. раздел «Статус: текущее vs целевое»).
>
> **Заметка будущему Claude/ревьюеру.** Прежде чем предлагать изменения — прочитай раздел «Принятые решения и обоснования» и «Чего НЕ строим». Многие «очевидные» альтернативы (event sourcing, кодоген всех сцен, состояние в payload кнопки) уже рассмотрены и сознательно отклонены под конкретные боли. Не возвращай их без новой причины.

## 1. Что такое Pangea

Текстовая RPG в виде чат-бота. Игрок взаимодействует через кнопки и текстовые сообщения; бот отвечает сообщениями. Сейчас интерфейс — ВКонтакте, в коде уже заложен и Telegram. Игровой цикл: регистрация → выбор расы → подземелье/лабиринт → случайные события (находки, бой) → инвентарь, статы, прокачка.

По своей сути это **конечный автомат над графом сцен** с доменной механикой (генерация лута, бой, статы) и случайностью.

## 2. Технологический стек

| Слой | Технология |
|---|---|
| Язык | Scala |
| Эффекты/конкурентность | ZIO (`Task = ZIO[Any, Throwable, A]`) |
| БД | PostgreSQL |
| Доступ к БД | doobie |
| JSON | circe |
| Перечисления | enumeratum (+ `DoobieEnum`) |
| Миграции | goose (SQL) |

DI — через конструкторы + `ZLayer` (сервисы инжектятся как поля, **не** через ZIO-окружение `R`). Этой конвенции придерживаемся везде.

## 3. Принципы (полярная звезда)

Семь обязательств, по которым проверяется любое изменение:

1. **Движок в центре, интерфейс — адаптер.** Игровая логика не знает про ВК. ВК/Telegram/БД/тесты — сменные адаптеры вокруг ядра.
2. **Контент — данные, механика — код.** Тексты, кнопки, переходы описываются декларативно; код остаётся только там, где есть логика (эффекты, доменные вычисления).
3. **Functional core / imperative shell.** Доменные решения (бой, лут, статы) — чистые функции без IO. Эффекты-shell их оркеструют: загрузил → позвал ядро → сохранил → отрендерил.
4. **Детерминизм через явный RNG.** Случайность — это вход (seed), а не скрытый `new Random()`. Ядро воспроизводимо; роллы пишутся в журнал.
5. **Три вида состояния — три хранилища.** Durable game state, live scene state, append-only журнал не смешиваются.
6. **Надёжность чат-бота by construction.** Сериализация действий на игрока + идемпотентность апдейтов — структурно, а не «по соглашению».
7. **Граф валидируется на старте.** Битые переходы, недостижимые узлы, тупики ловятся при загрузке, не у игрока.

## 4. Слои архитектуры

```
domain/   — чистое ядро: модели, бой, лут, статы, RNG (без ZIO)
engine/   — модель сцен, интерпретатор, роутинг Input
content/  — данные сцен + валидатор графа
ports/    — Renderer, Players, Journal, Clock, Rng (интерфейсы)
adapters/ — vk/ (вебхук + рендерер), postgres/ (репо, журнал), telegram/, test/
app/      — сборка ZLayer, вебхук-сервер, диспетчер
```

Зависимости направлены внутрь: `adapters` → `ports` ← `engine` → `domain`. Ядро ни от чего наружу не зависит.

### 4.1. Доменное ядро (`domain`)

Чистые функции, без ZIO/IO, RNG передаётся явно:

```scala
final case class Rng(seed: Long) { def next: (Long, Rng) = ??? }

def resolveCombat(hero: Hero, enemy: Enemy, rng: Rng): (CombatResult, Rng)
def rollLoot(level: Int, rng: Rng): (Item, Rng)
def applyLevelUp(hero: Hero): Hero
```

Тестируется property-тестами (HP не уходит в минус, лут уважает уровень). `ItemGenerator` — прообраз этого слоя; его надо очистить от скрытого `new Random()` и принимать `Rng` на вход.

### 4.2. Движок сцен (`engine`)

Платформо-независимое представление экрана и ввода:

```scala
final case class Screen(text: String, choices: List[Choice])
final case class Choice(id: ChoiceId, label: String, style: Style = Default)

sealed trait Input
object Input {
  case class Tap(choice: ChoiceId) extends Input  // нажал кнопку
  case class Say(text: String)     extends Input  // ввёл текст
}
```

Один интерпретатор обходит граф; per-state матчей `action` и enum'ов `Action` на сцену больше нет.

### 4.3. Порты и адаптеры (`ports` / `adapters`)

```scala
trait Renderer { def show(p: PlayerId, s: Screen): Task[Unit] } // ВК / Telegram / тест
trait Players  { def load(id: PlayerId): Task[PlayerState]; def save(s: PlayerState): Task[Unit] }
trait Journal  { def append(e: GameEvent): Task[Unit] }
```

ВК-адаптер превращает `Screen` в сообщение с клавиатурой и парсит входящий апдейт в `Update(playerId, Input)`. Telegram — второй адаптер той же формы, без дублирования логики.

## 5. Модель сцены

Узел графа — один из трёх видов:

- **Нарратив** — текст + одна кнопка «дальше» → следующий узел. (Линейные катсцены, как цепочка Travel в регистрации.)
- **Выбор** — текст + несколько кнопок, каждая ведёт в свою цель.
- **Эффект** — выполняет код и по результату выбирает следующий узел.

Цель кнопки (`Target`) — либо данные, либо код:

```scala
sealed trait Target
object Target {
  case class Goto(state: SceneId)                              extends Target // данные → в YAML/ресурс
  case class Run(effect: (User, UserAction) => Task[StateType]) extends Target // код (эффект)
}
```

**Граница данные/код проходит по `Target`.** `Goto` сериализуем (можно держать в данных), `Run` несёт функцию (эффект — всегда код). Сцена — «в основном данные с явными дырками под код».

**Правила для эффектов:**
- Сервисы (api, dao, repo) **захватываются** (эффект — метод стейта с этими полями), а не передаются в сигнатуру.
- Транспорт декодится **на границе**: эффект получает доменное значение (`Race`), а не `UserAction`. Хелпер `Run.decoded[A]` декодит payload и отдаёт чистый тип.
- В shell остаётся оркестрация (вызвать ядро, сохранить, отрендерить, выбрать следующую сцену) — это его законная работа, её не «прячут».

## 6. Модель состояния

| Вид | Что | Хранилище | Жизненный цикл |
|---|---|---|---|
| **Durable game state** | статы, инвентарь, прогресс | мутабельные таблицы (`heroes`, `inventory`) | источник истины |
| **Live scene state** | текущая сцена + scratchpad (выкаченный предмет до клика) | одна колонка-блоб на герое (`scene_data JSONB`) | перезаписывается следующей сценой |
| **Журнал** | что произошло (роллы, исходы, ошибки) | append-only таблица, партиционированная | retention-окно |

**Payload кнопки несёт только ссылку** (какая сцена / какой выбор), не состояние. У ВК payload ограничен по размеру; состояние живёт на сервере (live scene state), в кнопке — крошечный идентификатор.

Инвариант «ровно одно живое состояние на игрока» гарантирован **структурно** (одна колонка одной строки героя), а не соглашением.

## 7. Детерминизм и RNG

Игра полна случайности (`rollLoot`, выбор события, бой). RNG — инжектируемый seed на вход чистого ядра. Это даёт:
- тестируемость ядра без моков;
- воспроизводимость багов: seed и реальные роллы пишутся в журнал → сессию можно переиграть как было.

Запрет: скрытые `new Random()` внутри доменной логики.

## 8. Надёжность чат-бота

Чат-платформы передоставляют сообщения, игроки жмут дважды, апдейты приходят внахлёст. Решаем структурно:

- **Сериализация по игроку.** Per-user мейлбокс/семафор → переходы упорядочены, не гоняются. By construction убивает класс «два конкурентных перехода» (бывшие дубли `InProgress` событий).
- **Идемпотентность.** Каждый апдейт несёт id → дедуп по «последний обработанный апдейт на игрока», либо эффекты делаются идемпотентными. Дабл-тап не ломает стейт.

## 9. Журнал и партиционирование

Журнал — **append-only**: только `INSERT`, никаких `status`/`UPDATE`/`.unique`. Строка — это факт, который случился.

```sql
CREATE TABLE event_log (
    id BIGSERIAL, user_id BIGINT NOT NULL,
    event_type VARCHAR(64) NOT NULL, payload JSONB NOT NULL,
    trace_id UUID, occurred_at TIMESTAMP NOT NULL DEFAULT now(),
    PRIMARY KEY (id, occurred_at)          -- ключ партиции обязан входить в PK
) PARTITION BY RANGE (occurred_at);
```

- Партиции по месяцу; `DEFAULT`-партиция как предохранитель; нарезка на пару месяцев вперёд.
- Ретеншн = `DROP TABLE` старой партиции (мгновенно), а не `DELETE`.
- Обслуживание: **pg_partman (+pg_cron)**, если хостинг даёт расширения; иначе ZIO-джоб «создать вперёд / дропнуть старое». Решает доступность расширений на сервере.
- `trace_id` связывает строки одного действия в цепочку.
- Запросы — с границей по `occurred_at`, иначе нет partition pruning.

Логировать стоит **роллы и исходы** (что выкатил генератор, влез ли предмет, что выбрал игрок, ошибки) — то, что иначе не восстановить.

## 10. Жизненный цикл апдейта

1. ВК-вебхук → адаптер парсит в `Update(playerId, Input)`.
2. Диспетчер отдаёт в **сериализованный по игроку** обработчик.
3. Загрузка игрока (durable + текущая сцена + scratchpad).
4. Глобальная команда (`/home`, `/stats`) → обработать; иначе `Tap` → найти выбор в текущей сцене → выполнить `Target` (`Goto`/эффект).
5. Эффект: вызовы ядра + персист + `journal.append`, возвращает следующую сцену.
6. Рендер сцены через `Renderer` → ВК-адаптер шлёт сообщение.
7. Сохранить позицию + scratchpad; пометить апдейт обработанным.

## 11. Тестирование

- **Ядро** — property-тесты (инварианты боя/лута/статов).
- **Граф сцен** — headless-драйвер: «бот-игрок» ходит по графу через `Input`/`Screen` без ВК; целые нарративные ветки проверяются юнит-тестами.
- **Журнал** — проверка, что в ключевых точках падают правильные события.
- **Старт** — валидатор графа как тест: нет битых переходов/недостижимых узлов.

## 12. Статус: текущее vs целевое

Легенда: ✅ есть · 🟡 частично/временно · 🔴 нет · 🎯 цель

| Область | Статус | Комментарий |
|---|---|---|
| Базовый `State`-трейт, `StateHandler`, бутстрап юзера/героя | ✅ | Хороший фундамент, не трогаем |
| Стек ZIO/doobie/Postgres/circe, DI через конструкторы | ✅ | Конвенция зафиксирована |
| Сцены как классы (`DungeonState`, `RegistrationState`…) | 🟡 | Работает, но контент в коде; цель — данные + движок |
| Движок сцен (нарратив/выбор/эффект) | 🔴🎯 | Прототип `Narrative` сделан; нужен `Branch` + интерпретатор |
| Линейный нарратив как данные (цепочка Travel) | 🟡 | Прототип-рерайт `Registration` готов, не вкачен |
| Контент вынесен из кода | 🔴🎯 | Сейчас строки в Scala; цель — ресурсные файлы |
| Functional core (чистое ядро + явный RNG) | 🟡 | `ItemGenerator` близок, но со скрытым `Random` |
| Порты/адаптеры (Renderer и т.д.), ВК как адаптер | 🔴🎯 | Сейчас ВК протекает в стейты |
| Три вида состояния разведены | 🔴🎯 | Сейчас `events` смешивает live-state и журнал |
| Live scene state на герое (`scene_data`) | 🔴🎯 | Заменяет start/get/endEvent |
| Append-only журнал + партиционирование | 🔴🎯 | Текущая `events` не журнал |
| Сериализация по игроку + идемпотентность | 🔴🎯 | Латентные баги дабл-тапа |
| Валидатор графа на старте | 🔴🎯 | — |
| Headless-тесты флоу | 🔴🎯 | Сейчас флоу почти не тестируем |
| Кодоген клавиатур/Action (`kbgen`) | 🟡 | Вспомогательный инструмент; полезен, пока контент не уехал в данные |

## 13. Путь миграции (инкрементально, без переписывания всего)

1. **Движок рядом с `State`.** Ввести `Scene`/интерпретатор за существующим `State`-интерфейсом, не меняя `StateHandler`/схему БД.
2. **`Registration` первой.** Цепочка Travel → линейный нарратив-данные (прототип готов). Максимум удаляемого кода при изолированном риске.
3. **`Dungeon`, `HeroStats`, выбор расы** → узлы-выборы (`Branch` + `Goto`/`Run`).
4. **`FoundItem`** → `onEnter`-эффект + кнопки-`Run`; логика инвентаря не меняется.
5. **Состояние событий.** Завести `scene_data` на герое (live) и `event_log` (append-only журнал); вывести из обращения `events`-таблицу с её `.unique`/`status`.
6. **Очистить ядро.** `ItemGenerator` и будущий бой — на явный `Rng`.
7. **Вынести интерфейс в адаптер.** Ввести `Screen`/`Input`/`Renderer`; ВК-специфику убрать из стейтов.
8. **Контент в ресурсы.** Тексты по ключам → файлы; i18n как бонус.
9. **Надёжность.** Per-user сериализация + дедуп апдейтов.

Порядок такой, что каждый шаг — самостоятельный PR с откатом, и ранние шаги дают наибольшее упрощение.

## 14. Принятые решения и обоснования

| Решение | Почему | Что отклонено |
|---|---|---|
| Сцены как данные + интерпретатор | Контент в коде (Travel1..9) — 10 почти одинаковых методов; вставка сцены = renaming | Кодоген всех сцен в Scala (сотни файлов против одного движка) |
| Эффект указывает на цель напрямую (`Run`) | `Action.X => method` — 4 прыжка, непрозрачно; enum-тег врёт (`Travel` = «подтвердили расу») | Сохранение per-state Action-enum |
| Сервисы захватываются, не передаются | Эффект — метод стейта с полями-зависимостями; сигнатуру не раздуваем | Тащить сервисы в ZIO-окружении `R` (расходится с DI проекта) |
| Декод транспорта на границе | `selectRace`, дёргающий `ua.payload`, смешивает wire и логику | Эффекты, принимающие `UserAction` |
| Live state на герое + append-only журнал | `events` с `.unique` + всегда-INSERT — латентный краш на дублях; `endEvent` без фильтра статуса | Одна `events`-таблица как live-state и лог одновременно |
| Явный RNG | Скрытый `new Random()` не воспроизвести; баг рандома не отладить | RNG внутри доменных функций |
| Движок в центре, ВК — адаптер | ВК протёк в стейты; флоу нетестируем; Telegram дублировался бы | Логика, завязанная на `Keyboard`/`sendMessage` |
| Партиционирование журнала сразу | Append-only растёт быстро; ввести партиции потом — больно | `DELETE` по дате (бloat + vacuum) |

## 15. Чего НЕ строим (YAGNI)

- **Event sourcing.** Журнал наблюдательный; истина — в мутабельных `heroes`/`inventory`. Полный ES — оверкилл.
- **Микросервисы.** Один сервис + Postgres держит большую аудиторию.
- **Свой DSL контента на старте.** Сначала данные (HOCON/ресурсы) + валидатор; DSL — когда заболит.
- **Распределённая сериализация игроков.** Внутрипроцессной хватит, пока не упрёшься.
- **Кодоген всего.** `kbgen` — временный помощник; целевой путь для flow — интерпретатор, а не генерация Scala.

## 16. Боевая механика и формулы

### Урон игрока
```
damage = (atk + str) × spread / 100 × weaponMod × concBonus
```
- `atk` — `fightStats.atk`, растёт от оружия и баффов
- `str` — `baseStats.str`, растёт от прокачки
- `spread` — равномерное случайное значение 80–120 (разброс ±20%)
- `weaponMod` — 0.5 без оружия, 1.0 с оружием
- `concBonus` — `1 + concentration × 0.005`

**Важно:** формула аддитивна (`atk + str`), не мультипликативна. Умножение (`atk × str`) давало квадратичный рост и было исправлено.

### Броня
```
maxArmor = equipment.allArmor × max(fightStats.defence, 1)
```
- `equipment.allArmor` — сумма `armor` всех надетых предметов
- `defence` — из предметов; при `defence = 0` множитель равен 1 (броня работает без вложений в защиту)
- В бою броня поглощает урон первой и расходуется; на отдыхе восстанавливается до `maxArmor`

### Предметы и их ID
Каждый предмет получает реальный `BIGSERIAL` ID в таблице `items` **в момент генерации** (через `ItemRepository.generate` / `ItemRepository.persist`), до попадания в инвентарь. В JSON-блобе инвентаря и слотах экипировки хранится предмет с уже настоящим ID. Это гарантирует уникальность при `removeItem`.

### Отдых
После истечения `restDurationMs` (30 с — обычный, настраивается `DeathState`) восстанавливаются:
- HP до `effectiveMaxHp` (`vit × 16 × raceFactor`)
- Броня до `maxArmor`

### Боевой экран
Во время боя игроку показывается:
```
⚔ <Редкость Раса>
Раса: <раса> | ❤ <текущий>/<макс> | 🧥 <броня>/<макс> | ⚔ <урон>
🎯 Попадание: <шанс попасть по игроку>% | 💨 Уворот: <шанс увернуться от игрока>%

🧙 Вы: ❤ <текущий>/<макс> 🧥 <броня>/<макс>
🧪 Фляга: <заряды>
```
- **Попадание** — `mobHitChance(agi, evasion, battle)`, зажато в [5%, 95%]
- **Уворот** — `1 − playerHitChance(accuracy, battle)`, то есть вероятность, с которой моб уйдёт от удара игрока; зажато в [5%, 95%]
- Оба числа считаются из **эффективных** характеристик героя (с учётом баффов/дебаффов раунда)
- Броня показывается как `текущая/максимальная` и для моба, и для героя

### Фляга
```
healAmount = max(effectiveMaxHp / 4, 1)   // +25% от максимума за заряд
newHp      = min(currentHp + healAmount, effectiveMaxHp)
newCharges = flaskCharges - 1
```
- **Максимум зарядов:** `Hero.MaxFlaskCharges = 8`; ограничение фиксировано в коде и в БД (`CHECK (flask_charges <= 8)`)
- **Заряды пополняются** до максимума у источника воды (`DungeonState.healAtSpring`) и при создании нового героя
- **Один раз за раунд:** после выпитого заряда монстр наносит ответный удар (`monsterAttacks`); исключений нет
- Фляга должна быть экипирована в соответствующий слот; без экипировки UseFlask возвращает ошибку

## 18. Глоссарий

- **Сцена (Scene/State)** — узел графа: что показать и какие переходы. Бывает нарратив, выбор, эффект.
- **Бит (Beat)** — одна реплика линейного нарратива: текст + кнопка «дальше».
- **Эффект (Effect)** — код, выполняемый по кнопке: оркестрирует ядро/сервисы и выбирает следующую сцену.
- **Цель (Target)** — куда ведёт кнопка: `Goto` (переход-данные) или `Run` (эффект-код).
- **Live scene state / scratchpad** — переходное состояние текущей сцены (блоб на герое).
- **Durable game state** — постоянные данные игрока (статы, инвентарь).
- **Журнал (event_log)** — append-only лог фактов для дебага/аналитики.
- **Functional core / imperative shell** — чистое доменное ядро vs эффект-оркестратор.
- **Renderer / Adapter** — порт показа `Screen` и его реализация под платформу (ВК/Telegram/тест).
- **trace_id** — идентификатор, связывающий записи журнала одного действия.
